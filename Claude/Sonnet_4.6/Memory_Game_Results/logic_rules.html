01 Grid & Layout Math
Card grid dimensions
Math ▼
// Card count decision PAIRS = 8 TOTAL_CARDS = PAIRS × 2 = 16 // Grid layout: find integer factors of 16 // Options: 1×16, 2×8, 4×4, 8×2, 16×1 // Chosen: 4×4 — most square, fits screen COLS = 4 ROWS = TOTAL_CARDS / COLS = 4 // Card aspect ratio CARD_W = 3 CARD_H = 4 RATIO = CARD_H / CARD_W = 1.333 // portrait // Max grid width constrained to 560px GRID_MAX_W = 560px GAP = 16px // 1rem CARD_MAX_W = (GRID_MAX_W - GAP × (COLS-1)) / COLS = (560 - 48) / 4 = 128px max per card

The 4×4 grid was chosen because 16 cards is enough pairs to create genuine memory challenge without exhausting the player. A 4×4 grid is the most spatially balanced factoring of 16 — closer to square means less eye travel. The 3:4 portrait aspect ratio mimics a physical playing card, giving each card enough vertical space to show both an emoji and a text label.

Why 8 pairs, not 6 or 10
Constraint ▼
// Cognitive load estimate per pair count pairs = 6 → 12 cards, 4×3 grid // too easy, non-square pairs = 8 → 16 cards, 4×4 grid // ✓ balanced, square pairs = 10 → 20 cards, 4×5 grid // harder, non-square pairs = 12 → 24 cards, 4×6 grid // too hard for demo // Miller's Law: working memory holds ~7±2 items // At 8 pairs, player tracks max 8 positions = within limit MILLER_CAPACITY = 7 ± 2 // chunks in working memory PAIRS_CHOSEN = 8 // at upper edge of comfortable

8 pairs puts the game at the comfortable upper edge of working memory capacity (Miller's Law). 6 pairs would be solvable with very little memory effort. 10+ pairs would require external note-taking to play reliably, which conflicts with the accessibility goal of not requiring fine motor precision for note-taking.

02 Shuffle Algorithm
Fisher-Yates shuffle
Algorithm ▼
// Fisher-Yates (Knuth) shuffle — O(n), unbiased function shuffle(arr) { a = [...arr] // copy, don't mutate for i = a.length-1 → 1: j = floor(random() × (i+1)) // j ∈ [0, i] swap a[i] ↔ a[j] return a } // Input: 16 cards (8 pairs × 2) // Each permutation equally likely: 16! possibilities 16! = 20,922,789,888,000 // ~2.09 × 10¹³ arrangements // Why NOT naive sort-by-random: // arr.sort(() => Math.random()-0.5) ← BIASED, avoid

Fisher-Yates is the standard unbiased shuffle. Each of the 16! ≈ 20 trillion possible arrangements has an equal probability. A naive sort-by-random-key is biased because comparison-based sorts don't visit all swap positions equally — some arrangements end up more likely than others. With a memory game, a biased shuffle could make certain card positions predictably easier to find.

Food placement (no-overlap guarantee)
Algorithm ▼
// Snake game: place food not on snake body do { pos = { x: floor(random() × COLS), y: floor(random() × ROWS) } } while (snake.some(s => s.x===pos.x && s.y===pos.y)) // Probability of collision on first try: // P(hit) = snake_length / total_cells // At start: P(hit) = 3/441 ≈ 0.0068 (0.68%) // Near end (snake=200): P(hit) = 200/441 ≈ 0.45 // Expected iterations: 1/(1-P(hit)) — well bounded

The do-while loop retries placement until a free cell is found. This is a rejection sampling approach. The probability of needing many retries is low at game start but rises as the snake grows. In the worst practical case (~half the grid filled), expected retries are around 2. The loop terminates in finite expected time for any non-full grid.

03 Game State Logic
Match detection predicate
Logic ▼
// Two cards match if and only if their emoji is identical function isMatch(cardA, cardB): return cardA.emoji === cardB.emoji // NOT matched by id (each card has unique id) // NOT matched by label (labels are same when emojis match anyway) // Emoji chosen as key because it's the visible identity // Win condition: isWon = matched.length === cards.length = matched.length === 16 = all 8 pairs found // Guard conditions before flip allowed: canFlip = !locked && !flipped.includes(index) && !matched.includes(index)

Match detection uses strict equality on the emoji string. Each emoji is unique per pair, so this is a clean identity check. Three guard conditions prevent invalid flips: the locked flag blocks input during the mismatch timeout, the flipped check prevents re-clicking the same card, and the matched check prevents interacting with already-solved cards.

Progress counters
Math ▼
// moves: increments on every second flip (each "turn") moves += 1 // when flipped.length === 2 // matches displayed: pairs found, not individual cards matches_shown = matched.length / 2 = 0 → 8 // shown to user as "0–8" // remaining pairs: remaining = TOTAL_PAIRS - matched.length / 2 = 8 - matched.length / 2 = 8 → 0 // minimum possible moves to win (perfect play): MIN_MOVES = PAIRS = 8 // every flip is a match // theoretical max moves (worst case): // unbounded in theory, but practically ~40-60 for random play

Moves count "turns" not individual card flips — a turn is always exactly 2 flips. This is the standard scoring convention for memory games. Displaying matches as pairs (÷2) rather than individual cards is more meaningful to the player. Remaining counts down from 8 to 0 to give a clear sense of progress toward completion.

04 Timing Rules
Mismatch flip-back delay
Constraint ▼
// After mismatch: cards flip back after delay MISMATCH_DELAY = 1200ms // 1.2 seconds // Why 1200ms? // Too short (<600ms): player can't read both cards before they hide // Too long (>2000ms): game feels sluggish, breaks flow // 1200ms: enough time to read + mentally store both cards // CSS flip animation duration: FLIP_DURATION = 450ms FLIP_EASING = cubic-bezier(0.4, 0, 0.2, 1) // Mismatch delay >> flip duration ensures card is fully // visible before hiding: 1200 > 450 ✓ // aria-live announce delay (screen readers): ANNOUNCE_RESET = 50ms // clear then set, forces re-read

The 1200ms lock window was chosen to be long enough for a screen reader to announce the mismatch message and for a sighted user to mentally note both card positions. The 450ms CSS flip uses a Material Design standard easing curve (cubic-bezier 0.4, 0, 0.2, 1) which feels physical without being slow. The 50ms aria-live reset trick forces screen readers to re-announce identical messages.

Win banner display delay
Constraint ▼
// Win banner appears after last match animation completes WIN_DELAY = 600ms // Must be > flip animation: 600 > 450 ✓ // Gives player a moment to see the final match before // the win banner overlays the board // Focus management: move focus to "Play Again" button FOCUS_DELAY = 100ms // after banner renders in DOM

The 600ms win delay lets the final card's flip animation complete and gives the player a brief moment of satisfaction seeing the full matched board before the win overlay appears. The 100ms focus delay ensures the DOM has rendered the win banner before programmatic focus is moved to the Play Again button — important for screen readers and keyboard users.

05 Color & Contrast Ratios
WCAG contrast formula
Color ▼
// WCAG 2.1 relative luminance formula function luminance(r, g, b): // r,g,b ∈ [0,1] sRGB = c ≤ 0.03928 ? c/12.92 : ((c+0.055)/1.055)^2.4 return 0.2126×R + 0.7152×G + 0.0722×B function contrastRatio(L1, L2): return (max(L1,L2) + 0.05) / (min(L1,L2) + 0.05) // WCAG thresholds: // AA normal text: ratio ≥ 4.5:1 // AA large text: ratio ≥ 3.0:1 // AAA normal text: ratio ≥ 7.0:1

One honest failure: the muted text (#7c6f5e on #faf8f5) only achieves 3.8:1 — which passes AA for large text (≥18pt or bold ≥14pt) but fails for normal body text. This is used only for secondary labels and hints, never for critical game information. High contrast mode eliminates this entirely by replacing all muted text with #cccccc on black (12.6:1).

06 Animation Math
CSS 3D card flip geometry
Math ▼
// Card flip uses CSS 3D transform // Back face: rotateY(0deg) → visible at start // Front face: rotateY(180deg) → hidden at start (backface-visibility:hidden) // On flip: parent rotates 180° flipped : transform: rotateY(180deg) // Result: // back face: 0 + 180 = 180° → now hidden // front face: 180 + 180 = 360° ≡ 0° → now visible // Perspective depth (affects 3D "depth" appearance): perspective = 800px // Perceived depth ratio (d/perspective): // card width ~128px → 128/800 = 0.16 → subtle, not dramatic // Easing: cubic-bezier(0.4, 0, 0.2, 1) // t=0: velocity=0 (ease in) // t=0.5: max velocity (mid-flip is fastest) // t=1: velocity=0 (ease out, lands softly)

The flip works by rotating the parent container 180° around the Y axis. Since the front face starts pre-rotated 180°, after the parent rotates it lands at 360° = 0°, making it face forward. The 800px perspective value was chosen so the card width (≈128px) represents about 16% of the perspective depth — enough to feel 3D without looking like a dramatic barrel roll. The easing peaks at mid-flip, making it feel like a physical card being turned.

07 Accessibility Logic
aria-label state machine
Logic ▼
// Each card's aria-label transitions through 3 states: // State 1: face down (initial) label = `Card ${i+1} of 16, face down` // State 2: flipped (after click) label = `Card ${i+1}, ${card.label} ${card.emoji}, face up` // State 3: matched (permanent) label = `Card ${i+1}, ${card.label} — matched!` // State 2→1 (mismatch): reset after MISMATCH_DELAY label = `Card ${i+1} of 16, face down` // aria-live announcer messages: // flip: "Flipped {label}." // match: "Match! {label} pair found. {n} pairs remaining." // miss: "No match. {A} and {B} don't pair." // win: "Congratulations! You matched all pairs in {n} moves!"

The aria-label system ensures every state transition is communicated to screen readers. Three distinct label templates map to the three visual states a card can be in. The announcer region is separate from the labels — labels describe the card's current state when focused, while the announcer provides real-time event narration so the player always knows what just happened without needing to re-navigate to a card.

Focus management rules
Logic ▼
// Focus is moved programmatically at 3 moments: // 1. New game start → first card setTimeout(() => cards[0].focus(), 100ms) // 2. Win banner appears → Play Again button setTimeout(() => playAgainBtn.focus(), 100ms) // 3. Toolbar toggles → announce then stay on button // (no focus move needed — button retains focus) // Tab order: natural DOM order = grid reading order // Card 1(top-left) → Card 2 → ... → Card 16(bottom-right) // This matches visual left-to-right, top-to-bottom reading // Focus visible: browser default outline preserved // (never suppress outline without replacement)

Focus management follows three rules: move focus to the most logical next action after significant state changes, never suppress the browser's focus indicator, and ensure DOM order matches visual reading order so Tab navigation feels spatially coherent. The 100ms delays on programmatic focus moves give the DOM time to finish rendering before focus is applied — without these delays, focus may land on elements not yet fully visible.
